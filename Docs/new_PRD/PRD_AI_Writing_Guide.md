# AI를 활용한 PRD 작성 가이드

> PRD 문서를 AI(Claude, ChatGPT 등)의 도움으로 작성할 때 활용할 수 있는 프롬프트와 작성 요령입니다.

---

## 1. 기본 원칙

### 1.1 AI에게 맥락을 충분히 제공하기

AI는 프로젝트를 모릅니다. 항상 **배경 정보**를 먼저 제공하세요.

```
✅ 좋은 예:
"우리는 주식 포트폴리오 리밸런싱 앱을 만들고 있어.
React Native + Spring Boot 기술 스택이야.
Phase 1에서는 기본 CRUD와 리밸런싱 기능만 구현해.
지금 Phase 2 알림 기능 PRD를 작성하려고 해."

❌ 나쁜 예:
"알림 기능 PRD 작성해줘"
```

### 1.2 우리 템플릿 형식을 알려주기

AI에게 **원하는 출력 형식**을 명확히 지정하세요.

```
"다음 템플릿 형식으로 작성해줘:

## 요약 ⚡
- (3-5줄 핵심)

## Phase 2 (현재)
### 기능 목록
### 규칙

## Phase 3+ (확장 고려사항)
- [P3] ...

## 관련 문서
"
```

## 2. 상황별 프롬프트 예시

### 2.1 새 기능 PRD 작성

```markdown
# 프롬프트

나는 주식 포트폴리오 리밸런싱 앱의 PRD를 작성하고 있어.

## 프로젝트 배경
- 기술스택: React Native (Expo) + Spring Boot + MySQL
- Phase 1 완료: 구글 로그인, 포트폴리오 CRUD, 종목 관리, 기본 리밸런싱
- 현재 Phase 2 기능을 정의하려고 해

## 작성할 기능
[기능명]: 포트폴리오 공유 기능

## 요구사항 (내가 아는 것)
- 포트폴리오를 링크로 공유할 수 있음
- 공개/비공개 설정 가능
- 공유된 포트폴리오는 읽기 전용

## 작성해줘
다음 형식으로 PRD를 작성해줘:

1. 요약 ⚡ (5줄 이내)
2. Phase 2 기능 목록 (체크리스트)
3. 상세 규칙 (표로)
4. API 엔드포인트 (필요한 것만)
5. DB 변경사항 (필요한 컬럼)
6. Phase 3 확장 고려사항

Phase 마커 사용: [P2], [P3], [P3-고려]
```

### 2.2 기존 PRD 재구성

```markdown
# 프롬프트

아래는 우리 기존 PRD 중 일부야:

---
[기존 PRD 내용 붙여넣기]
---

이것을 다음 새 구조로 재구성해줘:

## 요약 ⚡
(핵심만 3-5줄)

## Phase 1 (현재)
(위 내용 중 P1에 해당하는 것)

## Phase 2+ (확장 고려사항)
(P2 예정인 것, 현재 설계 시 고려할 점)

## 관련 문서
(필요한 참조 링크)

규칙:
- 중복 제거
- Phase 마커 명확히
- 불필요한 설명 축약
```

### 2.3 Phase 2/3 세부 스펙 작성

```markdown
# 프롬프트

## 배경
우리 앱의 Phase 1 기능:
- 포트폴리오 CRUD (최대 5개)
- 종목 관리
- 기본 리밸런싱 제안

Phase 2에서 추가할 기능: **소수점 거래 지원**

## 질문
1. 소수점 거래를 지원하려면 기존 스키마에서 무엇을 변경해야 해?
2. 프론트엔드에서 입력 검증은 어떻게 해야 해?
3. 백엔드 계산 로직에서 주의할 점은?
4. Phase 3에서 확장할 수 있는 방향은?

## 출력 형식
PRD 형식으로 정리해줘. Phase 마커 사용.
```

### 2.4 누락된 부분 찾기

```markdown
# 프롬프트

아래는 우리 인증 기능 PRD야:

---
[기존 PRD 내용 붙여넣기]
---

이 PRD에서 빠진 것이 있는지 검토해줘:
- 보안 관점
- 에러 처리 관점
- 사용자 경험 관점
- Phase 2/3 확장성 관점

빠진 항목이 있으면 추가해서 완성된 버전을 보여줘.
```

## 3. 작성 시 체크리스트

### 3.1 요약 섹션 체크

```markdown
✅ 체크리스트:
- [ ] 5줄 이내인가?
- [ ] 핵심 기능이 한눈에 보이는가?
- [ ] 주요 제약사항이 포함되어 있는가?
- [ ] Phase 마커가 있는가?
```

### 3.2 Phase 마커 체크

```markdown
✅ 체크리스트:
- [ ] 모든 기능에 [P1], [P2], [P3] 마커가 있는가?
- [ ] [P2-고려] 마커로 현재 설계에 영향 주는 것을 표시했는가?
- [ ] P2/P3 내용이 너무 장황하지 않은가? (힌트만)
```

### 3.3 관련 문서 링크 체크

```markdown
✅ 체크리스트:
- [ ] DB 스키마 링크가 있는가? (reference/db-schema.md#테이블명)
- [ ] API 스펙 링크가 있는가? (reference/api-spec.md#엔드포인트)
- [ ] 관련 기능 문서 링크가 있는가?
```

## 4. AI 출력 검토 및 수정

### 4.1 AI가 자주 하는 실수

| 실수 | 해결 |
|------|------|
| 너무 장황함 | "더 간결하게 줄여줘" |
| 일반론만 | "우리 프로젝트에 맞게 구체화해줘" |
| Phase 혼재 | "P1/P2/P3 명확히 분리해줘" |
| 기술 스택 불일치 | 배경 정보에 기술 스택 명시 |

### 4.2 검토 프롬프트

```markdown
# 검토 요청 프롬프트

아래 PRD를 검토해줘:

---
[작성한 PRD]
---

검토 기준:
1. 요약이 5줄 이내인가?
2. Phase 마커가 일관적인가?
3. P1에 너무 많은 기능이 있진 않은가?
4. P2/P3 확장 고려사항이 현실적인가?
5. 빠진 에러 케이스가 있는가?

수정이 필요하면 수정된 버전을 보여줘.
```

## 5. Phase 2/3 작성 가이드

### 5.1 Phase 2 작성 시

Phase 2는 **P1 출시 후 가까운 미래**입니다.

```markdown
## Phase 2 작성 원칙

1. P1에서 이미 DB/코드에 **예약해둘 것** 명시
   예: "[P2-고려] is_public 컬럼 P1에서 미리 추가"

2. P1과의 **차이점** 위주로 작성
   예: "P1은 정수만, P2는 소수점 4자리까지"

3. **구현 힌트** 포함
   예: "calculateRebalancing() 함수를 인터페이스로 분리 권장"
```

### 5.2 Phase 3 작성 시

Phase 3는 **먼 미래 또는 가능성**입니다.

```markdown
## Phase 3 작성 원칙

1. 상세 스펙보다 **방향성**만
   예: "[P3] AI 기반 종목 추천 - 별도 마이크로서비스로 구현 예정"

2. P1/P2 구현에 **영향 없는 것**만
   예: "[P3] 현재 구현과 무관, 참고만"

3. **비용/리스크** 언급
   예: "[P3] 외부 API 연동 필요, 비용 발생"
```

## 6. 프롬프트 템플릿 모음

### 템플릿 1: 새 기능 초안

```
우리 [앱명] 앱의 [Phase N] [기능명] PRD를 작성해줘.

배경:
- 기술스택: [...]
- 완료된 기능: [...]

요구사항:
- [요구사항 1]
- [요구사항 2]

형식: 요약(5줄) → Phase N 상세 → 확장 고려사항 → 관련 문서
Phase 마커 사용: [P1], [P2], [P3]
```

### 템플릿 2: 기존 문서 재구성

```
아래 PRD를 새 형식으로 재구성해줘:

[기존 내용]

새 형식:
- 요약 ⚡ (5줄)
- Phase별 분리
- 확장 고려사항
- 관련 문서 링크
```

### 템플릿 3: 검토 및 보완

```
아래 PRD에서 빠진 것 찾아서 보완해줘:

[PRD 내용]

관점: 보안, 에러처리, UX, 확장성
```

## 7. 주의사항

### ✅ DO

- AI에게 **배경 정보** 충분히 제공
- **원하는 형식** 명확히 지정
- AI 출력 후 **직접 검토**
- **팀 컨벤션**에 맞게 수정

### ❌ DON'T

- AI 출력을 검토 없이 그대로 사용
- 배경 정보 없이 "~해줘"만 요청
- 너무 긴 프롬프트 (AI도 혼란)
- 한 번에 모든 것 요청 (단계별로)
